Why separated? Prevents queue contention. ACT_RU_* (hot) vs ACT_HI_* (cold/archive) can be on different storage tiers. 5 job table types avoid lock contention between immediate, scheduled, and suspended jobs.

The 54-Table Reality Check

❌Wrong Assumption
"54 tables × N microservices = huge overhead"
This assumes every microservice gets its own embedded Flowable with its own DB schema. This is an anti-pattern.
✅The Reality
"54 tables ONCE per environment — shared by all squads"
One Flowable cluster serves all teams. Processes isolated by tenantId. 3 environments × 54 = 162 tables total for the entire bank.
Analogy: You don't deploy a separate PostgreSQL or Kafka instance per microservice. Same principle — Flowable is a shared platform service. Like Kafka, it scales horizontally, not by multiplication.

Deployment: Newgen iBPS vs Flowable

Newgen iBPS Architecture
DB: Proprietary schema — similar table count, vendor-managed
Scaling: JBoss/WAS clustering — complex, expensive
Changes: Studio → test cycle → admin deploy → restart
Flowable Architecture (Recommended)
DB: Open-source schema — self-managed, horizontally scalable
Scaling: Kubernetes — same as your other Spring Boot services
Changes: Edit BPMN → push → hot-redeploy in <30s. No IDE.
Both are centralised — Newgen iBPS and Flowable both use a shared DB schema (similar table count). The key difference: Flowable's schema is open, self-managed, and cloud-native. iBPS schema is proprietary, vendor-supported, and tied to a J2EE server.

Resilience — Persist Then Execute

How It Works
1
Process reaches async task
2
Job written to ACT_RU_JOB (persisted)
3
Async executor picks up + executes job
4
Job removed on success. Process continues.
💥
Server crashes at step 3?
✓
Restart → executor retries job automatically
5 Dedicated Job Tables
ACT_RU_JOBImmediate async execution queue
ACT_RU_TIMER_JOBScheduled jobs (SLA timers, reminders)
ACT_RU_SUSPENDED_JOBPaused jobs (process suspended)
ACT_RU_DEADLETTER_JOBFailed after max retries — alert ops
ACT_RU_EXTERNAL_JOBExternal worker tasks (polling model)
Separating job types eliminates DB lock contention. Timer jobs and immediate jobs never compete for the same row locks.
