6 Capabilities in This POC

⚡
Parallel Async Processing
3 external screenings run simultaneously on separate threads. Flowable's async executor manages the thread pool — no custom concurrency code needed.
🔄
Hot Process Deployment
Deploy a new version of the BPMN in under 30 seconds with zero downtime. In-flight applications continue on v1; new applications start on v2. Both coexist.
👤
Human Task Management
Manual review tasks route to compliance and relationship-manager groups automatically. SLA boundary timer escalates to Senior Manager after 4 hours.
🛡️
Fault Tolerance & Resilience
Async tasks persist to database before execution. Server crash mid-process? On restart, Flowable's job executor picks up exactly where it left off — zero lost applications.
📋
Full Audit Trail
Every step, variable change, decision outcome, task assignment, and timer event is automatically recorded. HISTORY_LEVEL=FULL for regulatory compliance — no extra code.
🤖
Rules-Based Decisions
Decision engine applies R-001/R-002/R-003 policy rules. In production: replace with Flowable DMN decision tables — business-friendly spreadsheet rules that analysts can edit.

Under the Hood

54 Tables — Designed with Purpose

ACT_GE_* General

2
Binary blob store + engine properties. Shared by all modules.

ACT_RE_* Repository

3
Immutable deployments + process definitions. Enables v1/v2 coexistence.

ACT_RU_* Runtime

13
Live execution state. Rows deleted on completion — stays small. 5 dedicated job tables prevent queue contention.

ACT_HI_* History

10
Append-only audit trail. Can be moved to a separate read-replica for reporting without affecting runtime performance.

ACT_ID_* Identity

8
Built-in user/group store. Replaceable with Keycloak, LDAP, or Azure AD — drop these tables entirely.

ACT_DMN_* Decisions

5
DMN 1.2 decision tables. Business analysts can own credit policy rules as spreadsheets — no developer needed.

ACT_FO_* Forms

4
Form definitions + submitted data. Versioned alongside processes.

FLW_* Event Registry

6
Event-driven process triggering via Kafka, RabbitMQ, JMS. + Batch operation management.
Key insight: Runtime tables are fast & small (rows deleted on process end). History tables are append-only (ideal for columnar read replicas). The separation is deliberate — not accidental complexity.
Capability Deep-Dive

Resilience & Fault Tolerance

In Flowable, every async service task follows a persist-then-execute pattern. No work is ever lost.

1. Engine reaches async service task
→
2. Writes job to ACT_RU_JOB table (committed)
→
3. Async executor thread picks up & executes
⚠️ What if the server crashes here? ↓
4. Server restarts — job still in ACT_RU_JOB
→
5. Async executor automatically retries
→
6. Process continues exactly where it left off ✓
🔁 Retry Policy
Configurable maxRetries with exponential backoff. After N failures, job moves to ACT_RU_DEADLETTER_JOB for ops review — never silently lost.
⚖️ Lock-Based Clustering
Multiple Flowable nodes claim jobs via optimistic locking on LOCK_OWNER_ + LOCK_TIME_. No message broker needed for horizontal scale.
🏦 Implications for RAKBANK
A KYC service going down doesn't lose applications. Zero lost business. Zero manual reconciliation. Zero "where did that application go?" incidents.
Capability Deep-Dive

Hot Process Deployment

How a process change flows from idea to production:
1

Business analyst edits BPMN

In Flowable Modeler (drag-and-drop) or directly in XML. E.g., add a new "Sanctions Enhanced Review" step between Fircosoft and Risk Assessment.

2

Deploy via REST API or file drop

POST /repository/deployments — Flowable validates, parses, and saves v2 of the process. Takes <2 seconds.

3

Zero downtime switchover

New applications automatically use v2. The 14 applications in-flight right now? They continue on v1 undisturbed — no migration scripts, no manual intervention.

4

Version counter confirms it

GET /api/onboarding/process-definition returns "version": 2. Rollback is POST /deploy with the v1 XML — takes 10 seconds.
