Why separated? Separation prevents queue contention. Runtime (ACT_RU_*) tables are hot — high read/write. History (ACT_HI_*) tables grow unbounded and can be archived to cold storage. Job tables (ACT_RU_JOB, ACT_RU_TIMER_JOB) are separated to avoid lock contention between immediate vs scheduled jobs.

The 54-Table Reality Check

❌
The Assumption (Incorrect)
"54 tables × N microservices = huge overhead"
This assumes every microservice gets its own embedded Flowable with its own DB schema. This is an anti-pattern, not how enterprises deploy BPM.
✅
The Reality (Correct)
"54 tables once per environment — shared by all"
One Flowable platform serves all squads. 54 tables exist once per environment (dev/staging/prod). Analogy: you don't deploy a separate PostgreSQL instance per microservice.
Key insight: Just like a shared message broker (Kafka) or API gateway — the BPM engine is a shared platform service. 3 environments × 54 tables = 162 tables total for the entire bank, not 54 per service.

Multi-Squad Deployment Architecture

Deployment Architecture: Newgen iBPS vs Flowable

Newgen iBPS Architecture
Architecture: Single central iBPS server. All teams connect to it.
DB schema: Proprietary tables. Similar footprint to Flowable's 54.
Scaling: JBoss/WAS clustering — complex, expensive.
Changes: Require Studio IDE + admin deployment + restart window.
Flowable Architecture (Recommended)
Architecture: Central Flowable cluster OR embedded — your choice.
DB schema: 54 standard tables. Open source. Self-manageable.
Scaling: Spring Boot / Kubernetes — same as your other services.
Changes: Edit BPMN XML → push → hot-redeploy in <30s. No IDE lock-in.
Key difference: Both Newgen iBPS and Flowable use a centralised DB schema approach (similar table count). The difference is that Flowable's schema is open-source, self-managed, and horizontally scalable — while iBPS's schema is proprietary and tied to vendor support.
